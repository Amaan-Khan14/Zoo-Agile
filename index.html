The brief

Your team has been hired to build a zoo

The city has funded a new zoo and handed you an empty plot. Your team draws it into existence on a shared whiteboard — boundary wall, animal zones, toy train, aqua park, toilets, safety boxes, medical centre, the lot. The backlog holds 50 things to draw. You cannot draw all of them in 90 minutes. Two sprints to build a zoo that can actually open its gates.

What "done" means here

Everything is drawn, not written. A story is done when the thing it describes is actually drawn on the zoo map — the shape, the label, and every detail the acceptance criteria ask for. A sticky note saying "lion enclosure built" is not done. A drawn enclosure with a moat, two lions, viewing glass and a keeper door is done.

Rough shapes and stick figures are fine. Neatness earns nothing. Missing details cost points.

The trap: teams draw the shark tunnel and the toy train first because they are fun. Then the review asks where the boundary wall, the toilets and the medical centre are. A zoo with a dolphin show and no perimeter is not a zoo.

Constraints

  • Only the Product Owner may change priority or scope.
  • Sprint scope is frozen once planning ends. New ideas go to the backlog.
  • Every developer must own at least one story per sprint.
  • Work happens only on the shared whiteboard — if it is not drawn, it does not exist.
  • Draw on one single zoo map. Do not start a second map halfway through.
  • Velocity from Sprint 1 must be used to plan Sprint 2.

Setup

Run it with a shared data file

A browser cannot write to a folder on its own — that is blocked for security. So this folder ships with a tiny Python server. Start it once and the page genuinely reads and writes zoo-data.json in this same folder, every two seconds, with no clicking.

Start the server

  1. Keep index.html and server.py in the same folder.
  2. Open a terminal in that folder.
  3. Run python server.py
    On Mac/Linux use python3 server.py
  4. The browser opens by itself at http://localhost:8000
  5. The bar at the top of the page turns green and says Saved. That is the file being written.
Nothing to install. Python's built-in server is all it uses.

Let the team see it live

The server prints a second address, something like http://192.168.1.7:8000. Anyone on the same wifi can open that link on a laptop or phone.

  • Everyone sees the same board, the same backlog, the same acceptance criteria.
  • Changes appear on other screens within about 4 seconds.
  • Type your name in the box on the sync bar so saves are attributed.
Same wifi only. If the team is remote, share the whiteboard link and let one person drive this board.

Where the data lives

your-folder/ index.html the app server.py run this zoo-data.json <-- THE DATABASE backups/ zoo-data-20260722-1431.json zoo-data-20260722-1433.json (last 20 saves)

zoo-data.json is plain text. Open it in Notepad, email it, put it in Drive, commit it to git. Every save also writes a timestamped backup, so a bad move never costs you the session.

If you cannot run Python

The page still works fully — it just will not auto-save. Use the two buttons on the sync bar:

  • Download JSON — saves the current board to your Downloads folder.
  • Load JSON — restores it, on any machine, in any browser.

Download once at the end of each sprint and you never lose a review or a retro.

Between two teams? Download the JSON, then hit Reset board before the next group starts.

Kanban board

Track every story

Drag cards across the columns, or use the ← → buttons. Tag each card with an owner and a sprint. Mirror this on the whiteboard — this is the facilitator's copy so nothing gets lost when the board gets messy.

Rules of the board: only the card owner moves their own card. Nothing enters Sprint Backlog after planning ends. Nothing reaches Done until the PO has pointed at every acceptance criterion on the map.

The board lives in this browser tab only — it resets if you reload. Print it before you close the session.

Drawing kit

Everything is drawn

This is not a documentation exercise. Every story on the backlog describes a physical thing that has to appear on one shared zoo map. Rough shapes and stick figures are fine — the review only checks whether the details the story asked for are actually there.

The one rule

A box with the word "aviary" written inside it is not an aviary.

An aviary is a shape with a net line over the top, a double door, five birds and three trees inside — because that is what the acceptance criteria asked for.

At review the PO points a finger at each criterion on the map. If they have to describe what it would look like instead of pointing at it, the story is rejected.

Draw the legend first

Before anything else, agree one symbol for each repeated element and put them in a legend box in the corner. Then use the same symbol everywhere.

Split the map

Assign each developer a region during planning — animal zones, visitor services, back-of-house. Stops three people drawing over each other.

Draw big things first

Boundary wall, main path and service road before any enclosure. Teams that draw the shark tunnel first end up with no room for the vet hospital.

Name every card owner

Write the developer's name on the card and next to the drawn area, so the review knows who demos it.

Review & scoring

How the sprint gets judged

Sprint review scorecard (2 min)

The PO walks each Done card and scores it live.

  • Accepted — every criterion is visible on the map. The PO points at each one. Full points.
  • Partial — the thing is drawn but a detail is missing (moat drawn, keeper door missing). Half points, rolls to next sprint.
  • Rejected — not drawn, or only labelled as text. Zero points, back to backlog.
The PO must physically point at the drawing for each criterion. If they have to explain what it would look like, it is not accepted.

Bonus +10 pts: if all 10 Critical stories are Accepted by the end of Sprint 2, the zoo opens. If not, the zoo fails its licence inspection regardless of total points.

Velocity worksheet

The teaching moment is almost always the gap between committed and accepted in Sprint 1.

Retrospective (2 min) — start, stop, continue

Improvement commitment (2 min)

Pick exactly two changes to carry into Sprint 2. Two, not five — a team that commits to everything commits to nothing.

Facilitator: push back on vague commitments like "communicate better". Ask what would be visibly different on the board.
Facilitator notes

Running the session

Before the room fills

  1. Create one Web Whiteboard board per team and pre-draw the five columns plus the build canvas.
  2. Paste the backlog into the Product Backlog column, sorted by priority.
  3. Draw an empty rectangle on the right side of the board and label it THE ZOO. That is the map — teams draw inside it.
  4. Share the board link with all 5 members with edit rights.
  5. Open this page on the projector and keep the timer visible all session.

What to watch for

  • Over-commitment in Sprint 1. Let it happen. It is the lesson.
  • The PO who codes. Redirect them to acceptance and priority.
  • Silent developers. Enforce one owned story each — and one drawn area each.
  • Teams that write instead of draw. A box with the word "aviary" in it is not an aviary. Reject it at review once and they will never do it again.
  • Two people drawing over each other. Assign map regions during planning.
  • Scope creep mid-sprint. Point at the parking lot.
  • Retro that lists 9 improvements. Force it down to two.

Debrief questions (if you have spare minutes)

  • What did you commit to in Sprint 1, and what did you actually finish? Why the gap?
  • Who decided what was in scope, and did the process hold under time pressure?
  • Which acceptance criteria were argued about at review? Would a sharper AC have prevented that?
  • Look at the map. Could a real visitor walk from the gate to every animal and back out? Could an ambulance reach every zone?
  • Did the two retro commitments change anything measurable in Sprint 2?
  • If there were a Sprint 3, what would you pull first — and would it be the same answer you gave 90 minutes ago?
